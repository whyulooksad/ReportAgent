# 整体架构设计

Multi_agent_graph.py里构建了一个多智能体协作架构，通过LangGraph构建了一个有向图，包含多个节点，每个节点代表一个智能体或一个处理步骤。整个流程如下：

1. 用户输入一个自然语言目标（goal）。
2. 通过main_question智能体解析用户目标，生成一组自然语言查询（plan）并判断意图（meaning）。
3. 逐个执行这些查询（通过NL2SQL智能体），并收集结果。
4. 根据意图（meaning）决定是直接返回查询结果（meaning为"query"）还是生成一份报告（meaning为"report"）。

### **状态管理 **

```
class GraphState(TypedDict):
    goal: str                    # 用户原始需求
    plan: List[str]             # 待执行的查询列表
    current_query: Optional[str] # 当前执行的查询
    results: List[str]          # 执行结果集合
    iterations: int             # 执行计数器
    done: bool                  # 完成标志
    final_report: str           # 最终输出
    all_queries_snapshot: List[str] # 查询快照
    meaning: Optional[str]      # 意图分析：query/report/other
```

作用：在整个工作流中传递和共享数据。

### **智能体服务配置**

```
MAIN_QUESTION_URL = "http://localhost:8000/generate"  # 意图分析智能体
NL2SQL_URL        = "http://localhost:8001/nl2sql"    # 查询执行智能体  
MAIN_REPORT_URL   = "http://localhost:8002/api/generate-report" # 报告生成智能体
```

### workflow节点详解

#### 节点1: question_node

- 调用 `main_question` 智能体

- 解析返回结果，提取 用户意图`meaning` (query/report/other)
- 提取具体的查询计划 `plan`

#### 节点2: pick_next_query_node

- 从plan中按序取出一个NL查询，放入current_query;
- 若plan为空，则标记done=ture

#### 节点3: exec_nl2sql_node

- 读取 current_query，调用 NL2SQL智能体；
- 将查询与结果拼装成一个文本片段，追加到 results；

#### 节点4:finalize_report_node

- 若 meaning == "query"：不生成报告，直接汇总 NL2SQL 结果作为最终输出；

- 若 meaning == "report"：调用main_report智能体生成报告；

# NL2SQL智能体：自然语言转SQL

docs文件夹下存放表字段的详细解释；hf_models文件夹下是一个普通的embedding模型；knowledge_graph文件夹下是知识图谱的相关代码；rag文件夹下是整个NL2SQL智能体用到的所有向量库和相应的rag流程代码；schema_cache文件夹下是数据库结构缓存；schema_engine文件夹下是数据库结构解析引擎；get_schema_cache.py是数据库结构预热脚本；runtime_context.py实现时区隐式注入，为裸LLM提供时区信息；agent.py里是这个智能体的整体代码实现；agent_mod.py里实现了其api包装，满足后续分布式架构。

整体流程如下：

1. 分析查询需求：从用户的自然语言查询中识别出涉及的表和字段。
2. 使用图谱进行路径规划：识别出涉及的表和字段后，使用工具`find_join_path`查找这些表的字段的连接关系。
3. 获取字段解释：在生成 SQL 查询之前，使用工具`retrieve_field_docs` 获取表中涉及字段的详细解释。
4. 生成 SQL 查询：结合路径规划字段说明，生成一个 sqlserver 支持的有效 SQL 查询。
5. 执行 SQL 查询：生成的 SQL 查询将通过工具`execute_sql` 执行，并返回查询结果。
6. 过滤查询结果：在最后一次执行SQL查询得到输出结果后，使用工具`retrieve_rules_docs`获取过滤规则，过滤输出结果，并将过滤后的数据作为最终结果输出。

**find_join_path ：图谱路径规划**

```
@tool
def find_join_path(start_column: str, end_column: str) -> str:
    """
    在 Neo4j 知识图谱中查找起点列到终点列的最短路径，并返回涉及的表和结构说明。
    """
    sq = SchemaGraphQuery()
    path = sq.find_path(start_column, end_column)
    if not path:
        return "未找到路径"

    # 获取涉及的表
    tables = sq.extract_tables_from_path(path)

    # 返回路径节点和涉及的表结构说明
    return f"涉及的表: {tables}，路径节点: {[n['name'] for n in path.nodes]}"
```

设计理念：提前确定表间关联，解决复杂 join 问题。

实现详解:见知识图谱部分。

#### 向量库构建流程

**retrieve_field_docs实现**：

数据源：docs文件夹下的`schema_doc.md` 文档

元数据设计：

分块策略：按数据库表切分，每个表作为一个独立的文档块

```
def parse_schema_md_by_table(md_text: str) -> List[Dict]:
    """
    将 markdown 内容按表切块，每个表一个 chunk，内容为整个表的字段说明
    
    参数:
        md_text: str - 包含表结构说明的 Markdown 文本
    返回:
        List[Dict] - 每个元素为一个表的信息字典，包含:
            - content: 表的完整说明内容（表名+字段信息）
            - metadata: 元数据，其中包含表名（table 字段）
    """
    # 存储所有表的解析结果
    entries = []
    # 将输入的 Markdown 文本按行拆分，便于逐行处理
    lines = md_text.splitlines()

    # 临时存储当前正在处理的表名（初始为 None，表示未开始处理任何表）
    current_table = None
    # 临时存储当前表的内容行（收集表名行之后的字段说明等内容）
    current_chunk = []

    # 逐行遍历 Markdown 文本的每一行
    for line in lines:
        # 正则匹配表名行（格式如：### 表名：table_name 或 ### 表名: table_name）
        table_match = re.match(r'^### 表名[:：]?\s*([a-zA-Z0-9_.]+)', line)
        
        if table_match:
            # 若匹配到新表名，说明之前可能在处理另一个表
            # 检查是否有未保存的表内容（当前表名和内容不为空）
            if current_table and current_chunk:
                # 将之前收集的表内容封装成字典，添加到结果列表
                entries.append({
                    "content": "\n".join(current_chunk).strip(),  # 拼接内容行并去除首尾空白
                    "metadata": {"table": current_table}  # 记录对应的表名
                })
                # 重置当前内容列表，准备收集新表的内容
                current_chunk = []
            
            # 提取新表名（正则匹配到的分组内容）
            current_table = table_match.group(1)
            # 在当前内容列表中添加表名标识（便于后续查看）
            current_chunk.append(f"【表名】{current_table}")
        
        elif current_table:
            # 若未匹配到新表名，但正在处理某个表（current_table 不为空）
            # 则当前行是该表的字段说明内容，加入临时内容列表
            current_chunk.append(line)

    # 循环结束后，处理最后一个表（避免遗漏未保存的表内容）
    if current_table and current_chunk:
        entries.append({
            "content": "\n".join(current_chunk).strip(),
            "metadata": {"table": current_table}
        })

    # 返回所有表的解析结果
    return entries
```

def get_rules_by_knowledgecontent(query: str, score_threshold: float = 0.7) -> List[Document]:
	embedding = LocalBGEEmbeddings()
    db = Chroma(persist_directory=str(RULES_CHROMA_DB_DIR), embedding_function=embedding)
    # 使用带分数的相似度搜索
    docs_with_scores = db.similarity_search_with_score(query, k=5)  # 先获取较多结果
    # 根据置信度阈值过滤结果
    filtered_docs = []
    for doc, score in docs_with_scores:
        # 注意：Chroma 返回的是距离分数，越小表示越相似
        #将其转换为置信度 (1 - 距离)
        confidence = 1 - score
        if confidence >= score_threshold:
            # 将置信度添加到文档元数据中
            doc.metadata["confidence"] = round(confidence, 3) #保留最相近的3个
            filtered_docs.append(doc)
    # 只返回满足阈值条件的文档，如果没有则返回空列表
    return filtered_docs

#### 节点与关系设计

节点类型：

1. Table节点 - 代表数据库表

   ```
   Table节点属性：
   - name: 完整表名（schema.table格式）
   - comment: 表注释说明
   ```

2. Column节点 - 代表表字段

   ```

   - name: 完整字段名（schema.table.column格式）
   - type: 字段数据类型
   - comment: 字段注释说明
   ```

关系类型：

1. HAS_COLUMN关系 - 表与字段的包含关系

   双向关系：Table ↔ Column

   设计理念：支持从表找字段，也从字段找表

2. FOREIGN_KEY_TO关系 - 字段间的外键关联

   双向关系：Column ↔ Column

   来源：数据库原生外键 + 人工补充外键

    graph = Graph(NEO4J_URI, auth=(NEO4J_USER, NEO4J_PASSWORD))
    graph.delete_all()
    
    # 2. 创建表和字段节点
    for table, table_info in mschema_obj.tables.items():
        fq_table = qualify_table(table)  # 标准化表名
        table_node = Node("Table", name=fq_table, comment=table_info.get('comment', ''))
        graph.merge(table_node, "Table", "name")

        for field_name, field_info in table_info["fields"].items():
            fq_col = qualify_column(f"{table}.{field_name}")

            # 建立双向HAS_COLUMN关系
            graph.merge(Relationship(table_node, "HAS_COLUMN", col_node))
            graph.merge(Relationship(col_node, "HAS_COLUMN", table_node))

        fq_col2 = qualify_column(f"{table2}.{col2}")

核心查询方法：

```
def find_path(self, start_column: str, end_column: str):
    """
    查找两列之间的最短路径
    支持通过外键关系和包含关系进行路径探索
    
    参数:
        start_column: 起始列名
        end_column: 目标列名
    返回:
        Path对象或None（未找到路径时）
    """
    # 自动补全dbo前缀
    if not start_column.startswith("dbo."):
        start_column = f"dbo.{start_column}"
    if not end_column.startswith("dbo."):
        end_column = f"dbo.{end_column}"
    
    # Neo4j Cypher查询
    query = """
    #使用shortestPath算法寻找最短连接路径
    MATCH p = shortestPath(
    	#路径关系类型：FOREIGN_KEY_TO（外键）和HAS_COLUMN（包含），最大路径深度：5跳
        (start:Column {name: $start})-[:FOREIGN_KEY_TO|HAS_COLUMN*1..5]-(end:Column {name: $end})
    )
    RETURN p
    """
    result = self.graph.run(query, start=start_column, end=end_column).data()
    return result[0]['p'] if result else None
```
